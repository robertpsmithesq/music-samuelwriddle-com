# Technical PRD: Public Audio Clips from Cloudflare R2 in SvelteKit

## 1. Overview

Implement support for embedding public audio clips on the music/showcase website. Audio files will be stored outside the Git repository in Cloudflare R2 and served through a custom Cloudflare-managed media subdomain. The SvelteKit application will render audio players using static metadata stored in the repo.

This feature is intended for a showcase website, not a full streaming platform. The implementation should prioritize simplicity, maintainability, low cost, and clean presentation over advanced media-platform functionality.

## 2. Product Goal

Allow visitors to play selected public music clips directly from the website without relying on SoundCloud storage or third-party embed players.

The site should remain a presentation layer. Cloudflare R2 should provide object storage and media delivery. Railway should not serve or proxy audio files.

## 3. Non-Goals

This feature will not include:

- User accounts
- Private audio access
- Signed URLs
- DRM or download prevention
- Upload UI or admin dashboard
- Database-backed track management
- Server-side audio proxying
- Audio transcoding pipeline
- Waveform generation
- Advanced listener analytics
- Full streaming-platform behavior

## 4. Architecture

### Intended request flow

```txt
Visitor browser
  ↓
SvelteKit-rendered page
  ↓
<audio src="https://media.example.com/clips/demo-v1.mp3">
  ↓
Cloudflare edge/cache
  ↓
Cloudflare R2 bucket, only when needed
```

### Domains

Primary website:

```txt
example.com
www.example.com
```

Media domain:

```txt
media.example.com
```

Replace `example.com` with the actual site domain.

### Storage

Create one Cloudflare R2 bucket for public showcase media.

Suggested bucket name:

```txt
music-showcase-media
```

Suggested bucket structure:

```txt
clips/
  track-one-demo-v1.mp3
  track-two-demo-v1.mp3

artwork/
  track-one-cover-v1.jpg
  track-two-cover-v1.jpg
```

## 5. Cloudflare R2 Requirements

### 5.1 Bucket

Create an R2 bucket dedicated to public media assets.

The bucket should contain only files that are safe to expose publicly. Do not upload private demos, unreleased tracks, stems, project files, or source-quality masters unless they are intentionally public.

### 5.2 Public access

Expose the bucket using a custom domain:

```txt
media.example.com
```

Cloudflare supports connecting an R2 bucket to a custom domain from the bucket settings, under public access/custom domains. Custom domains are preferred over the default public R2 URL because they provide cleaner URLs and better control over caching/security behavior. Cloudflare documents custom domain setup for R2 public buckets.

### 5.3 Caching

Configure Cloudflare cache behavior for:

```txt
media.example.com/*
```

Recommended initial behavior:

```txt
Cache eligibility: Cache everything
Edge TTL: 1 month or longer
Browser TTL: 1 day to 1 week
```

Use versioned filenames when replacing audio files:

```txt
clips/song-demo-v1.mp3
clips/song-demo-v2.mp3
```

Avoid overwriting files at the same URL unless the Cloudflare cache will be purged.

### 5.4 CORS

Do not configure CORS for the first pass unless needed.

Native HTML audio playback through:

```html
<audio controls src="..."></audio>
```

does not usually require custom CORS configuration.

Add R2 CORS later if the app needs browser JavaScript to fetch, inspect, visualize, or process the audio file directly, such as for waveform rendering or Web Audio API features. Cloudflare provides R2 CORS configuration for public buckets.

Example future CORS policy:

```json
[
	{
		"AllowedOrigins": ["https://example.com", "https://www.example.com"],
		"AllowedMethods": ["GET", "HEAD"],
		"AllowedHeaders": ["*"],
		"ExposeHeaders": ["ETag"],
		"MaxAgeSeconds": 3600
	}
]
```

## 6. SvelteKit Requirements

### 6.1 Track metadata

Create a static metadata file in the SvelteKit project.

Suggested file:

```txt
src/lib/data/tracks.ts
```

Example:

```ts
export type Track = {
	slug: string;
	title: string;
	description: string;
	audioPath: string;
	artworkPath?: string;
	durationLabel?: string;
	externalLinks?: {
		soundcloud?: string;
		bandcamp?: string;
		youtube?: string;
		spotify?: string;
	};
};

export const MEDIA_BASE_URL = 'https://media.example.com';

export function mediaUrl(path: string): string {
	return `${MEDIA_BASE_URL}/${path.replace(/^\/+/, '')}`;
}

export const tracks: Track[] = [
	{
		slug: 'first-demo',
		title: 'First Demo',
		description: 'A short public demo clip.',
		audioPath: 'clips/first-demo-v1.mp3',
		artworkPath: 'artwork/first-demo-cover-v1.jpg',
		durationLabel: '0:45',
		externalLinks: {
			soundcloud: 'https://soundcloud.com/...'
		}
	}
];
```

### 6.2 Audio component

Create a reusable audio clip component.

Suggested file:

```txt
src/lib/components/AudioClip.svelte
```

Example:

```svelte
<script lang="ts">
	import { mediaUrl, type Track } from '$lib/data/tracks';

	export let track: Track;
</script>

<article class="track-card">
	{#if track.artworkPath}
		<img src={mediaUrl(track.artworkPath)} alt="" loading="lazy" class="track-card__artwork" />
	{/if}

	<div class="track-card__content">
		<h3>{track.title}</h3>

		{#if track.description}
			<p>{track.description}</p>
		{/if}

		{#if track.durationLabel}
			<p class="track-card__duration">{track.durationLabel}</p>
		{/if}

		<audio controls preload="metadata" src={mediaUrl(track.audioPath)}>
			Your browser does not support the audio element.
		</audio>

		{#if track.externalLinks}
			<div class="track-card__links">
				{#if track.externalLinks.soundcloud}
					<a href={track.externalLinks.soundcloud}>SoundCloud</a>
				{/if}

				{#if track.externalLinks.bandcamp}
					<a href={track.externalLinks.bandcamp}>Bandcamp</a>
				{/if}

				{#if track.externalLinks.youtube}
					<a href={track.externalLinks.youtube}>YouTube</a>
				{/if}

				{#if track.externalLinks.spotify}
					<a href={track.externalLinks.spotify}>Spotify</a>
				{/if}
			</div>
		{/if}
	</div>
</article>
```

Use:

```svelte
<audio controls preload="metadata">
```

rather than aggressively preloading full audio files. This helps keep the page lighter.

### 6.3 Page integration

Render the track list from metadata.

Example:

```svelte
<script lang="ts">
	import AudioClip from '$lib/components/AudioClip.svelte';
	import { tracks } from '$lib/data/tracks';
</script>

<section>
	<h2>Selected Clips</h2>
	<p>Short audio excerpts and demos.</p>

	<div class="track-list">
		{#each tracks as track}
			<AudioClip {track} />
		{/each}
	</div>
</section>
```

## 7. File Format Requirements

Initial supported file format:

```txt
.mp3
```

Recommended audio behavior:

- Use compressed MP3 files for public web playback.
- Keep clips short where possible.
- Avoid uploading WAV files for normal web playback.
- Use lowercase, hyphenated filenames.
- Include version numbers in filenames.

Good:

```txt
clips/silver-light-demo-v1.mp3
```

Avoid:

```txt
clips/Silver Light Final FINAL.wav
```

## 8. Security and Abuse Considerations

### 8.1 Public file assumption

All files in the public R2 bucket should be treated as publicly downloadable.

Do not rely on the HTML audio player to prevent downloads. If the browser can play the file, users can inspect network requests and access the file URL.

### 8.2 Keep Railway out of media delivery

The app must not proxy audio through SvelteKit endpoints or Railway.

Avoid:

```txt
/routes/api/audio/[track]/+server.ts
```

unless a future feature truly needs server-side media handling.

### 8.3 Cloudflare protections

Use Cloudflare as the front door for both the website domain and the media domain where feasible.

For the media domain, start with caching-focused protection rather than aggressive bot challenges, since native audio requests should not be interrupted.

During abuse or a suspected DDoS, possible emergency actions include:

- Temporarily disable public access to the R2 bucket
- Temporarily block `media.example.com/clips/*`
- Temporarily remove audio embeds from the site
- Add temporary Cloudflare WAF rules for abusive traffic patterns

## 9. Cost Management

Cloudflare R2 currently includes a free monthly allowance for storage and operations, and charges separately for Class A and Class B operations after included usage. Public reads are the main operation type to watch for this feature. R2 also advertises free internet egress, which makes it a reasonable fit for public static media compared with egress-metered storage providers.

Implementation requirements:

- Set Cloudflare billing alerts if available on the account.
- Set Railway usage/billing alerts if available.
- Confirm that audio requests go to `media.example.com`, not the Railway app.
- Confirm Cloudflare cache is being used for media requests.
- Keep an emergency shutoff plan documented.

## 10. Observability

Minimum useful checks:

- Verify audio files load from the public media domain.
- Verify browser devtools show audio requests going to `media.example.com`.
- Verify the SvelteKit app does not receive audio file requests.
- Check Cloudflare analytics for request volume after launch.
- Check Railway metrics after launch to ensure audio playback is not increasing app traffic.

Optional later:

- Cloudflare Web Analytics
- Plausible or similar privacy-friendly analytics
- Basic event tracking for play clicks
- R2 object inventory or scripted validation

## 11. Accessibility Requirements

Each audio clip should include:

- Visible track title
- Short text description or context
- Native browser audio controls
- Link text that clearly identifies external destinations

Artwork should use either:

```html
alt=""
```

for decorative images, or descriptive alt text if the artwork communicates meaningful information.

## 12. Acceptance Criteria

The feature is complete when:

- An R2 bucket exists for public media assets.
- The bucket is connected to a custom media domain.
- At least one MP3 file is uploaded under `clips/`.
- The uploaded file is publicly accessible at `https://media.example.com/clips/...`.
- The SvelteKit app renders at least one reusable audio component.
- The audio component plays the R2-hosted file using a direct media URL.
- Audio requests do not route through Railway.
- Track metadata is centralized in a typed `tracks.ts` file.
- File naming conventions are documented and followed.
- Cache behavior is configured for the media domain.
- There is a documented emergency shutoff procedure.
